# 별빛로그 업그레이드 계획 v1

> 작성일: 2026-06-09  
> 목표: 게임 업적 방식의 뱃지 & 칭호 시스템 도입으로 학생 참여도 향상

---

## 1. 개요

학생이 활동을 수행할 때마다 조건을 달성하면 뱃지를 획득하는 게임 업적 방식의 시스템.
뱃지를 모을수록 칭호가 업그레이드되어 학생이 지속적으로 앱을 사용하도록 동기부여.

---

## 2. 뱃지 시스템

### 2.1 기본 원칙

- 뱃지는 한 번 획득하면 영구 보유
- 조건 달성 순간 즉시 팝업으로 알림
- 미획득 뱃지는 흑백 + 자물쇠로 표시 (달성 조건 힌트 제공)
- 각 카테고리별로 난이도가 낮은 것부터 높은 순으로 구성

## 4. UI/UX 구성

### 4.1 학생 화면 구조

```
학생 메인 화면 상단
├── 프로필 영역: [이름] · [현재 칭호] · ★ X / 20
└── [내 뱃지 보러가기 →] 버튼

뱃지 모음판 페이지 (/student/badges)
├── 상단: 칭호 카드
│   ├── 현재 칭호 (크게 표시)
│   ├── 진행 바: ████████░░░░ X / 20개
│   └── 다음 칭호까지 N개 더 필요
│
└── 하단: 뱃지 모음판 (20칸 그리드)
    └── 뱃지 카드 × 20개 (모두 처음부터 배치됨)
        ├── 미획득: 흑백 + 흐릿하게 표시 + 자물쇠 아이콘 + 달성 조건 힌트
        └── 획득:  컬러 + 선명하게 + 빛나는 테두리 + 획득 날짜
```

### 4.2 뱃지 모음판 상세 디자인

#### 뱃지 카드 1개의 구성 요소
```
┌─────────────────┐
│   🌈 or 🔲      │  ← 아이콘 (획득: 컬러 이모지, 미획득: 흑백 필터)
│   감정 무지개    │  ← 뱃지 이름
│  6가지 감정 기록 │  ← 달성 조건 (항상 표시)
│  2026.06.09     │  ← 획득 날짜 (획득 시에만 표시)
│   🔒 or ✓       │  ← 상태 아이콘
└─────────────────┘
```

#### 흑백 → 컬러 전환 방식
- **미획득 상태**: CSS `filter: grayscale(100%) opacity(0.4)` 적용
- **획득 순간**: `filter` 제거 + `box-shadow: 0 0 16px [카테고리 색상]` 추가
- **전환 애니메이션**: 획득 시 0.5초 동안 흑백 → 컬러 페이드 인 (`transition: filter 0.5s`)

#### 카테고리별 컬러 테마
| 카테고리 | 테마 색상 | 뱃지 테두리 색 |

## 5. DB 스키마 변경

```sql
-- students 테이블에 컬럼 추가
ALTER TABLE students ADD COLUMN badge_count INT DEFAULT 0;
ALTER TABLE students ADD COLUMN title TEXT DEFAULT '별빛 새싹';

-- 학생 뱃지 획득 기록 테이블 (신규)
CREATE TABLE student_badges (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  student_id UUID REFERENCES students(id) ON DELETE CASCADE,
  badge_id TEXT NOT NULL,          -- ex. 'emotion_first', 'plan_perfect_5'
  earned_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(student_id, badge_id)     -- 뱃지 중복 획득 방지
);
```

## 7. 뱃지 달성 여부 확인 로직 상세

### 7.1 전체 흐름

```
학생이 액션 수행 (감정 기록 저장 등)
    ↓
API Route Handler (서버)
    ↓
1. 원래 액션 처리 (DB INSERT/UPDATE)
    ↓
2. checkAndAwardBadge(studentId, trigger) 호출
    ↓
3. 해당 트리거와 연관된 뱃지 목록 순회
    ↓
4. 각 뱃지: 이미 획득했는지 확인 → 조건 충족 여부 쿼리 → 지급
    ↓
5. 새로 획득한 뱃지 목록을 API 응답에 포함
    ↓
클라이언트에서 새 뱃지가 있으면 팝업 표시
```

---

### 7.2 뱃지 유형별 달성 확인 방법

#### 유형 A — 누적 횟수형
> 감정 탐험가(10회), 감정 수집가(30회), 감정 박사(100회), 작은 실천가(5회), 큰 실천가(30회) 등

```
액션 발생 시 → DB에서 해당 학생의 누적 횟수 COUNT 조회
→ 조건값(10, 30, 100…)과 비교
→ COUNT >= 조건값이면 달성
```

- 매 액션마다 COUNT 쿼리를 실행하는 **즉시 집계 방식** 사용
- 학급 규모(30명 내외)에서는 쿼리 부담이 거의 없음
- 카운터 컬럼을 별도로 유지하면 빠르지만 동기화 오류 위험이 있어 단순하게 집계 방식으로 처리

#### 유형 B — 첫 1회 달성형
> 첫 별빛, 첫 계획, 첫 성찰, 첫 편지

```
액션 발생 시 → student_badges 테이블에 해당 badge_id가 존재하는지 확인
→ 없으면 지급 (별도 COUNT 쿼리 불필요)
```

- 가장 단순한 로직: 뱃지가 없으면 지급하는 것 자체가 "첫 1회" 조건을 만족

#### 유형 C — 연속 기록형
> 7일의 기록 (7일 연속 감정 기록)

```
액션 발생 시 → 오늘 포함 최근 7일의 날짜 목록 생성 [D-6, D-5, ..., D-0]
→ emotions 테이블에서 해당 학생의 기록을 날짜(DATE)로 GROUP BY
→ 7개 날짜 모두 기록이 존재하는지 확인
→ 모두 있으면 달성
```

- 날짜는 한국 KST 기준으로 비교
- 연속이 끊겨도 뱃지를 회수하지 않음 (한 번 획득하면 영구 유지)

#### 유형 D — 다양성 달성형
> 감정 무지개 (6가지 감정 카테고리 모두 기록)

```
액션 발생 시 → 해당 학생의 전체 감정 기록에서 DISTINCT category 조회
→ 반환된 카테고리 집합이 전체 6개를 모두 포함하는지 확인
→ 포함하면 달성
```

#### 유형 E — 복합 조건형
> 완벽한 하루 (같은 날 감정 기록 + 계획 달성률 100% 동시 달성)

```
계획 완료 또는 감정 기록 시 → 오늘 날짜 기준으로 두 조건 동시 확인
  조건1: 오늘 감정 기록이 1건 이상 존재하는지
  조건2: 오늘 계획 전체가 완료 상태인지 (완료 수 / 전체 수 = 100%)
→ 둘 다 만족하면 달성
```

- 감정 기록 저장 시에도, 계획 완료 시에도 동일하게 체크 (어느 쪽이 나중에 완료될지 모름)

#### 유형 F — 달성률형
> 오늘도 실천, 작은 실천가, 큰 실천가 (계획 달성률 100% 누적)

```
계획 항목 완료 체크 시 → 오늘 해당 학생의 전체 계획 수와 완료된 계획 수 조회
→ 완료 수 / 전체 수 = 1.0 이면 오늘 100% 달성
→ plans 테이블을 날짜 + 학생 기준으로 GROUP BY해서 100% 달성 날짜 수 집계
→ 5회, 30회 조건과 비교
```

### 7.3 핵심 함수 구조 (pseudo-code)

```
function checkAndAwardBadge(studentId, trigger):
  1. BADGE_TRIGGER_MAP에서 trigger에 해당하는 뱃지 목록 가져오기
  2. 이미 획득한 뱃지 목록 조회 (student_badges WHERE student_id = studentId)
  3. 각 뱃지 순회:
     a. 이미 획득했으면 SKIP
     b. 뱃지 유형(A~F)에 따라 조건 확인 쿼리 실행
     c. 조건 충족 → student_badges INSERT, students.badge_count +1
     d. 칭호 업그레이드 여부 확인 (getTitleByBadgeCount) → students.title UPDATE
  4. 새로 획득한 뱃지 목록 반환

function getTitleByBadgeCount(count):
  0개   → '별빛 새싹'
  5개~  → '별빛 탐험가'
  10개~ → '별빛 기록자'
  15개~ → '별빛 마스터'
  20개  → '별빛 전설'
```
